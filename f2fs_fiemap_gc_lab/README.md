# Android F2FS FIEMAP GC Lab

This directory contains a host-side harness for testing whether Android package
artifacts move at the FIEMAP level during F2FS garbage collection.

Target files:

- `base.apk`
- split `*.apk`
- `base.odex`
- `base.vdex`
- explicitly supplied `--extra-path` files

The success condition for a clean GC relocation result is:

- file content hash stays the same,
- device/inode stays the same,
- FIEMAP physical extents change,
- F2FS moved-block or GC counters increase.

## No Android build required

Start with a probe. This only needs `adb` and root:

```sh
cd tools/f2fs_fiemap_gc_lab
python3 f2fs_fiemap_gc_lab.py --probe-only
```

The probe writes `probe_summary.txt` and `device_capabilities.json`. If the
device has `f2fs_io fiemap`, the harness uses that for exact-file FIEMAP
comparison. If `f2fs_io` is unavailable but `filefrag` exists, it falls back to
`filefrag`. If neither is available, the harness can still run donor pressure
and F2FS GC and record hashes, inodes, and GC counters, but it cannot prove that
a specific APK/ODEX/VDEX file moved.

## No-build Play Store workflow

Use this when the app is already installed:

```sh
python3 f2fs_fiemap_gc_lab.py \
  --package your.package.name \
  --gc-seconds 60 \
  --donor-count 64 \
  --donor-mib 4 \
  --collect-segment-info
```

This uses only shell-accessible mechanisms: package discovery, `stat`,
`sha256sum`, device `f2fs_io fiemap` or fallback `filefrag`, donor files, and
F2FS sysfs `gc_urgent`.

## No-build controlled APK install workflow

This is better when you have APK files locally and want higher probability. The
harness keeps donor files alive during install, then deletes them before GC.

```sh
python3 f2fs_fiemap_gc_lab.py \
  --package your.package.name \
  --install-apk /path/to/base.apk \
  --install-apk /path/to/split_config.apk \
  --gc-seconds 60 \
  --donor-count 64 \
  --donor-mib 4 \
  --collect-segment-info
```

For a single APK, provide one `--install-apk`. For split APKs, repeat
`--install-apk`; the harness uses `adb install-multiple -r`.

## Optional native helper

`f2fs_gc_probe.c` is still useful if you later can run a device binary. It adds
native `FS_IOC_FIEMAP`, ordinary F2FS GC ioctl, and F2FS GC range ioctl support.
It is optional for the no-build flow.

```sh
export ANDROID_NDK_HOME=/path/to/android-ndk
make android ANDROID_ARCH=arm64 ANDROID_API=30

python3 f2fs_fiemap_gc_lab.py \
  --package your.package.name \
  --helper ./f2fs_gc_probe.android \
  --range-gc \
  --gc-one-trials 128
```

## Output

Each run creates a local directory such as `f2fs-gc-lab-20260630-143000`.
Important files:

- `probe_summary.txt`: generated by `--probe-only`.
- `device_capabilities.json`: detected device-side FIEMAP/GC tools, including `f2fs_io`.
- `summary.txt`: quick verdict per target.
- `summary.json`: machine-readable result.
- `targets.txt`: files under test.
- `snapshots/before.json` and `snapshots/after.json`: stat, hash, FIEMAP data if available.
- `logs/f2fs_attrs_before.txt` and `logs/f2fs_attrs_after.txt`: sysfs counters.
- `logs/segment_info_before.txt` and `logs/segment_info_after.txt`: only when
  `--collect-segment-info` is used.

`summary.txt` uses these verdicts:

- `MOVED`: FIEMAP changed while hash/inode stayed stable.
- `same-map`: FIEMAP was available and did not change.
- `map-unk`: no exact FIEMAP source was available for that file.
